# 私域聊天记录人格与关系分析项目

> **状态**：🟡 v0.4 架构定稿（持续细化中）
> **发起日期**：2026-03-07
> **项目类型**：新项目探索 / 开源作品方向
> **工作代号**：待定

---

## 📌 一句话定义

一个基于 **私域聊天记录** 的 **AI + 心理学** 人格与关系分析助手：
通过对聊天记录进行结构化分析，帮助用户理解 **自己、对方与双方关系**，并输出 **可解释、可回溯、有证据锚点** 的沟通与互动建议。

---

## 🧭 当前总判断

**这个方向值得做，而且已经不是空想。**

当前调研收敛后的判断：

1. **需求真实存在。**
   用户确实需要“关系沟通副驾”这类工具，尤其在暧昧、关系修复、冲突沟通、边界表达、被冷落/被误解等高情绪高不确定场景中。

2. **市场已有验证，但产品普遍不够深。**
   已经有人在做“聊天记录 → 关系分析 / 回复建议 / 红旗识别”，但大多停留在一次性分析、轻度娱乐化或话术辅助层。

3. **真正的空位在于：**
   - 长期、对象级建模
   - 证据化分析
   - 科学心理学框架
   - 隐私友好
   - 持续更新的关系理解能力

4. **产品不能做成把 MBTI 宿命化、算命化的贴标签工具。**
   MBTI 本身不是星座式玄学，而是心理学脉络中的人格类型框架；问题不在 MBTI 本身，而在把它绝对化、神断化、配对宿命化的用法。项目可以使用 MBTI 作为入口和展示语言，但底层必须坚持更稳的心理学构念与行为证据。

---

## 🪧 项目气质 / 方法论宣言

> **我们是科学。心理学。**

这句话是本项目的核心约束。

因此本项目必须始终坚持：

1. **不把 MBTI 玄学化、宿命化。**
2. **不做宿命论人格配对。**
3. **不做读心术式结论。**
4. **不做操控他人的工具。**
5. **重要判断尽量给出证据与不确定性。**
6. **显式区分：稳定倾向 / 关系中的模式 / 当前状态。**

也就是说，本项目不是：
- AI 算命
- 把 MBTI 用成星座式娱乐算命
- “拿捏对方”的操控辅助
- 给第三方偷偷建档的监控工具

而是：
- 自我理解工具
- 关系分析助手
- 沟通副驾
- 私域关系决策支持系统

## 🎯 产品定位

### 产品角色定位
不是“操控他人心理工具”，而是：
- **关系沟通副驾**
- **自我剖析助手**
- **关系分析与复盘工具**
- **私域关系决策支持工具**

### 一个项目，双入口
本项目不拆成“自己项目”和“他人项目”两个独立产品，而是做成：

- **看懂自己**：偏自我剖析、成长建议、沟通盲区识别
- **看懂这段关系**：偏关系状态、对方互动倾向、下一步建议

### 核心报告结构
所有核心报告建议统一为三层：
- **我**
- **TA**
- **我们**

这样既能覆盖自我理解，也不会把“他人分析”做成脱离关系语境的贴标签工具。

## 🧱 方法论框架（当前共识）

## 1. 产品层原则：前台简单，后台复杂

当前最重要的产品判断：

> **复杂的方法论应藏在引擎里，前台只给用户一个简单、好懂、能代入、能行动的结果。**

也就是说：
- **前台输出** 应尽量简单直接
- **后台推理** 可以复杂，但不要把复杂性压到用户脸上

### 用户前台最需要看到的，其实是：
1. **我可能是什么类型**
2. **我现在处在这个类型的什么阶段**
3. **我的优势和短板在哪里**
4. **我怎么补劣势、怎么进阶**
5. **如果是关系场景，我和 TA 卡在哪里**

这比一开始端出一堆复杂维度更符合用户直觉。

---

## 2. MBTI 的产品定位

### 1) MBTI 不是星座式玄学，但不能被玄学化
当前共识：
- **MBTI 本身不是玄学**，而是心理学脉络中的人格类型框架
- **问题不在 MBTI 本身，而在把它宿命化、绝对化、神断化的用法**

所以项目不是反对 MBTI，而是反对把 MBTI 用成：
- 宿命论配对表
- 算命式人格断言
- 从少量聊天直接判死的真型神断

### 2) 前台可以以 MBTI 为主
V1 前台完全可以以 MBTI 为核心输出语言，因为它天然适合产品化：
- 用户熟悉
- 传播力强
- 代入感强
- 结构简单
- 容易形成“类型 + 阶段 + 建议”的结果页

### 3) 后台不能只靠 MBTI
为了让前台那张“可能类型 + 阶段 + 建议”的卡更靠谱，后台仍需要更稳的支撑：
- **行为信号**（聊天证据层）
- **Big Five**（稳定人格维度层）
- **Attachment / 依恋维度**（亲密关系风险层）
- **Communication Style / 沟通模式**（互动行为层）
- **Relationship State / 关系状态**（当前阶段层）

也就是说：
> **前台可以主打 MBTI，后台必须用更稳的证据与构念把它托住。**

---

## 3. 后台推理的基本原则

### 1) 证据优先
不能只输出“看起来像很懂”的报告。
所有重要判断都应尽量绑定：
- 聊天片段
- 行为模式
- 时间窗口
- 置信度 / 不确定性说明

### 2) 区分三种东西
报告里要显式区分：
- **稳定倾向**
- **关系中的模式**
- **当前阶段状态**

避免把“一时状态”误写成“永久人格”。

### 3) 允许用户校准
系统不应一锤定音，而应允许用户校准：
- 你觉得这个描述像不像你？
- 候选类型里哪个更贴近你的自我认知？
- 这段关系里你更像追击方还是退缩方？

这样能把产品从“AI 定义你”变成“AI 帮你理解和验证”。

---

## 🧠 MBTI 在本项目里的正确角色

### V1 前台推荐的核心输出
对于大多数用户，前台最适合直接给：

1. **可能类型**
   - 例如：你大概率是 INTP
   - 可附 1~2 个备选类型

2. **当前阶段**
   - 例如：低阶态 / 发展中 / 进阶态 / 压力失衡态
   - 或用更温和的说法：初阶 / 成长中 / 整合中 / 失衡中

3. **优势 / 短板**
   - 这个类型当前最明显的长板是什么
   - 在关系与沟通里最容易卡住的短板是什么

4. **劣势与补全方向**
   - 当前最薄弱的功能 / 表现面是什么
   - 应该怎样补、怎样进阶、怎样避免失衡

5. **关系场景下的卡点**
   - 如果是双人分析，再加：你和 TA 主要卡在哪里

换句话说，用户真正想看到的是：
> **可能类型 + 当前阶段 + 优势短板 + 进阶建议**

这比把一整套复杂心理学模型直接端给用户更符合产品直觉。

### 后台可以更复杂，但不该压给用户
为了让上面这张卡更靠谱，后台仍然可以使用：
- 行为信号
- 候选类型机制
- 认知功能映射
- Big Five / 依恋 / 沟通模式
- 证据与置信度

但这些更适合作为引擎与证据层，而不是首页主视图。

### 适合做成产品卡片的部分
- **可能类型卡**
- **当前阶段卡**
- **优势 / 短板卡**
- **进阶建议卡**
- **关系卡点卡**
- **备选类型 / 为什么像不像**

### 不适合做核心结论的部分
- 从少量聊天精确判死 16 型
- “你们天生绝配 / 天生相克”
- 阴影功能神断
- loop / grip 泛化解释一切
- 把 MBTI 当创伤、人格障碍或精神健康解释器
- 把“低阶 / 高阶”写成人格等级制或价值排序

## 🧾 输入设计

## A. 客观输入：聊天记录
优先支持：
- Markdown 聊天记录
- TXT / 结构简单的导出文本
- 聊天截图 / 长截图

候选来源：
- 微信聊天记录（经导出 / 转换后）
- QQ 聊天记录（经导出 / 转换后）
- Telegram / WhatsApp 等文本导出
- OpenAI / ChatGPT / OpenClaw / Cherry Studio 等 AI 对话记录
- 手机聊天截图、桌面聊天截图、超长滚动截图

### 当前原则
先统一成内部标准消息格式，再做分析。

### 新增判断：V1 需要考虑截图输入
现实里很多用户手上没有标准导出文本，只有：
- 若干聊天截图
- 长截图
- 拼接截图
- 局部裁剪图

因此输入层不能只考虑“文本导入”，还必须考虑：
> **图片里的聊天记录如何被恢复成结构化对话。**

也就是说，系统最前面应增加一层：
- **多模态输入解析层**

它不只是 OCR，而是要处理：
- 文本识别
- 聊天气泡边界识别
- 左右说话人归属判断
- 时间戳/系统提示识别
- 长截图顺序恢复
- 多张截图拼接与去重
- 图片解析结果转内部 message schema

---

## B. 主观 / 背景输入：用户主动补充信息
聊天记录之外，还应支持用户主动补充一些聊天中未必完整体现的信息，例如：
- 自己或对方已知 / 自测的 MBTI
- 双方身份与关系背景（同学、同事、暧昧对象、前任等）
- 关键上下文说明（异地、刚吵过架、长期未见面等）
- 用户当前目标（想推进、想修复、想确认边界、想复盘）
- 某些关键事件的补充说明

### 重要原则
报告中应明确区分：
- **来自聊天记录的证据**
- **来自用户主动提供的背景信息**

不能把二者混成同一种“事实”。

## 🔐 隐私路线（当前共识）

## 1. 当前结论
**V1 不强求完整 Local-only。**

原因：
- 纯本地模型能力与工程复杂度都较高
- 第一阶段重点是先把产品做出来、把方法论跑通
- 完整 local-only 成本过高，不适合作为最初门槛

因此更现实的路线是：

### 第一步：先用 BYOK 作为隐私第一步
- 用户使用自己的模型 Key 进行分析
- 产品本身尽量不托管统一 AI 账户池
- 先解决“不是把高敏感聊天直接交给一个黑箱第三方平台代管”的问题

### 第二步：后续补本地脱敏能力
后续可考虑引入：
- **0.8B 级别的小模型**
- 轻量规则系统
- 本地实体识别 / 匿名化 / 片段筛选

也就是先在本地做：
- 脱敏
- 实体替换
- 敏感字段清洗
- 片段筛选

再把必要内容发给云端大模型分析。

### 第三步：长期再探索更强本地能力
例如：
- local inference
- 端侧 embedding
- 端侧分类与信号抽取
- 更强的自部署能力

但这些不应阻塞第一版落地。

---

## 2. 隐私原则
1. **BYOK 不等于绝对隐私。** 必须如实说明云端模型的存在与边界。
2. **隐私不能靠承诺，最好靠默认系统设计保证。**
3. **默认最小上传、最小留存、可删除、可审计。**
4. **要考虑聊天另一方的隐私，而不只是当前用户。**
5. **本项目是帮助理解关系，不是帮助秘密监控别人。**

---

## 3. V1 隐私承诺方向
- 开源
- self-host / local-first 友好
- BYOK
- 不强求云账号体系
- 尽量减少黑箱感
- 后续逐步补本地脱敏

一句话总结：
> **第一版先用 BYOK 解决“信任与掌控”的第一步；后续再逐步补上本地脱敏与更强隐私能力。**

## 📤 输出设计

## 前台输出应该尽量简单直接
当前产品判断：
**不要把复杂方法论直接压给用户。**

第一眼最适合让用户看到的，是一套简单直观的结果：

### 1. 可能类型
- 你大概率是什么类型
- 可附 1~2 个备选类型

### 2. 当前阶段
- 低阶态 / 发展中 / 进阶态 / 压力失衡态
- 或更温和版本：初阶 / 成长中 / 整合中 / 失衡中

### 3. 优势 / 短板
- 你的天然强项是什么
- 你当前最容易吃亏的地方是什么

### 4. 劣势补全 / 进阶建议
- 你该补哪块
- 你在关系里该怎么练
- 你现在最应该避免什么低阶表现

### 5. 关系场景补充
如果是双人分析，再补：
- TA 可能是什么类型
- 你们分别卡在哪里
- 当前最核心的误解点或拉扯点是什么

---

## 证据与复杂解释应作为第二层
在第一层简单结果之下，再展开更细的内容：

### 第二层可包含
- 候选类型与为什么像 / 不像
- 聊天行为证据
- 当前关系状态
- 沟通模式解释
- 置信度与反证
- 来自聊天记录的证据 vs 用户补充背景信息

也就是说：
- **首页给结果感**
- **下钻页给证据感**

这更符合用户实际使用习惯。

## 🏗️ MVP 设想（第一版）

## MVP 目标
先完成：
> **输入聊天记录 + 背景信息 → 输出结构化人格 / 关系分析报告 + 沟通建议**

### 第一版建议能力
1. 上传 md / txt 聊天记录
2. 上传聊天截图 / 长截图（优先做静态图片，不做音视频）
3. 填写背景补充信息
4. 完成多模态输入解析、基础归一化与信号抽取
5. 输出结构化报告
6. 给出可操作建议
7. 对关键判断附证据锚点

### 第一版建议聚焦场景
优先做：
- 1 对 1 聊天分析
- 恋爱 / 暧昧 / 关系修复 / 自我剖析

### 第一版尽量不要做
- 音视频深度解析
- 全平台原生复杂备份兼容
- 重型多 Agent 编排
- 操控型建议
- 玄学配对结论

说明：
- **图片 / 截图输入已纳入 V1 范围**
- 但仅限“静态截图 → 结构化聊天恢复”
- 不扩展到视频、语音、屏幕录制等更重多模态

## 🧩 建议的系统链路

1. **聊天记录导入**
2. **数据归一化**
3. **行为信号抽取**
4. **人格与关系构念映射**
5. **MBTI 结构化解释层**
6. **建议生成**
7. **报告 IR / 卡片化输出**
8. **页面渲染与证据回查**

### 当前正式架构取向
- **单 pipeline** 优先
- 可插拔 reviewer / skeptic 模块
- 不把多 Agent 当 MVP 核心卖点
- **明确区分多模态输入解析层 / 规则壳 / Embedding 预处理层 / 子模型层 / 主模型层**

原因：
- 聊天关系建模前半段更像数据 pipeline
- 可解释性和稳定性比“炫酷讨论”更重要
- 便于 debug、复算、迭代指标体系
- 不同层对“输入恢复 / 机械处理 / 模糊预处理 / 局部理解 / 全局判断”各有最合适的承担者

### 当前正式分层架构

#### -1. 多模态输入解析层
负责把不同形态的输入先还原成“可分析对话”：
- 文本文件导入解析
- **VLM-first 图片解析**（默认，不单独建设传统 OCR 主链路）
- 聊天气泡 / 条目级内容恢复
- 左右说话人归属判断
- 时间戳 / 系统提示识别
- 长截图顺序恢复
- 多张截图拼接、去重、对齐
- 模型输出草稿转内部 message schema

原则：
- 不把截图直接扔给后续人格分析层
- 不要求模型一次性生成终态复杂 schema
- 先让模型输出最小结构化草稿，再由程序补全归一化

#### 0. 规则壳 / orchestration 层
负责整个流程的编排与可复算控制：
- 输入归一化
- 基础规则清洗
- 各层调用顺序控制
- 中间结果缓存
- 审计、重跑与调试

原则：
- 所有智能层都应被程序外壳包裹
- 不让 embedding 或模型“自己决定全部流程”

#### 1. Embedding 预处理层
负责“模糊机械活”的智能化：
- 语义向量化
- 相邻窗口语义距离计算
- 候选切片边界检测
- 相似片段聚类
- 近重复去重
- 候选高价值片段召回
- 模糊噪音片段初筛

定位：
- 不是最终判断层
- 而是**增强程序预处理能力**的基础设施层

原则：
- embedding 负责找、聚、召回、压缩
- 不直接负责人格功能判定

#### 2. 子模型层
负责局部片段理解：
- 片段级弱信号提取
- 候选功能提示
- 证据句抽取
- 片段级置信度判断
- 歧义点初步解释

原则：
- 只看局部片段
- 不负责最终类型结论
- 以快、便宜、结构化输出稳定为优先

#### 3. 主模型层
负责全局综合判断：
- 全局一致性审查
- 区分稳定人格 / 压力态 / 情境角色
- 处理冲突证据
- 输出 Top 3 类型候选
- 输出阶段、优势短板、进阶建议
- 生成最终报告

原则：
- 看全局，不做机械活
- 用更强模型做最后综合判断

一句话总结：
> **多模态输入解析层先把截图/文本还原成结构化对话，规则壳负责编排，Embedding 层负责语义预处理，子模型负责局部读片段，主模型负责最后综合判断。**

### 对 BettaFish / MiroFish 的路线映射
当前判断：这两个项目给了本项目一个很清晰的双阶段模板。

- **BettaFish / 微舆** 更偏“分析现在发生了什么”
- **MiroFish** 更偏“推演接下来可能会怎样”

映射到本项目中：

#### 第一阶段：先做我们的 BettaFish
也就是先把 **分析层** 做好：
- 你可能是什么类型
- TA 可能是什么类型
- 你们现在处在什么阶段
- 当前关系主要卡点是什么
- 各自优势短板是什么
- 该怎么沟通 / 修复 / 推进

这个阶段的目标是：
> **先看懂现在。**

#### 第二阶段：再考虑我们的 MiroFish
也就是以后再加 **预测 / 推演层**：
- 如果继续这样互动，关系大概率会怎么走
- 如果改变表达方式，结果可能怎样变化
- 如果现在推进 / 放缓 / 澄清，分别可能带来什么后果
- 冲突后的不同处理路径，各自风险点是什么

这个阶段的目标是：
> **再推演未来。**

### 当前结论
**第一版不要同时做分析 + 预测。**

原因：
- 预测层更容易飘、玄、不可验证
- 分析层才是可信度与产品口碑的基础
- 只有先把“看懂现在”做好，后面才有资格做“预测未来”

一句话总结：
> **先做我们的 BettaFish，再考虑我们的 MiroFish。**

## 🌱 初期策略

### 开源优先
当前阶段不以商业化为第一优先级，更适合先做成一个 **开源 GitHub 项目**。

原因：
- 当前身份适合先做作品、方法论和验证
- 这类产品涉及高敏感隐私，开源更容易建立信任
- 有利于社区传播、收集反馈、迭代输入格式与分析框架
- 先验证“有没有人愿意用”，再考虑“怎么收费”

### 当前更看重的“财富”
当前的长期价值不一定先来自付费，而可能先来自：
- 作品资产
- 认知资产
- 影响力资产
- 未来选择权

因此第一阶段更像：
> 一个可信、可自部署、可自带模型 Key 的开源分析工具

而不是先做成重运营、重变现的商业产品。

## ✅ 当前已确认的关键决策

1. 这是一个值得继续推进的新项目方向。
2. 不拆成两个项目，而是 **一个项目、双入口**。
3. 报告结构统一为 **我 / TA / 我们**。
4. **MBTI 适合做 V1 的结构化入口与展示层，不适合独占底层。**
5. 底层长期应转向 **行为证据 + Big Five + 依恋 + 沟通模式**。
6. 项目气质必须坚持：**科学、心理学、证据支持，不做玄学。**
7. 输入不仅包含聊天记录，也必须支持 **用户主动补充背景信息**。
8. **V1 输入范围正式包括：文本聊天记录 + 聊天截图 / 长截图。**
9. 初期不以变现为目标，优先做 **开源 + BYOK**。
10. V1 不强求完整 local-only，先用 BYOK 解决第一步隐私与掌控问题。
11. 后续可以考虑引入 **0.8B 小模型** 做本地脱敏与预处理。
12. 产品路线参考 BettaFish / MiroFish：**第一版先做分析层（我们的 BettaFish），不要急着做预测层（我们的 MiroFish）。**
13. **当前正式架构定为五层：多模态输入解析层 → 规则壳 → Embedding 预处理层 → 子模型层 → 主模型层。**
14. **Schema 设计原则定为：分层完整，但单条 message 保持瘦身，分析结果外置。**
